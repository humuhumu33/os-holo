The Q door

Q is the one door to hologram-native intelligence. Every verb runs on the device, seals a self-verifying PROV-O receipt, and — by Law L5 — refuses to fabricate a result rather than fake one. This page gives each verb’s exact signature, return shape, and refusal behaviour, so you (or an agent) can call it with no guessing.

Note

Receipts are re-derivable objects: every Q run links its inputs (prov:used) and output (prov:generated) so the answer can be audited, not just trusted.

Q.fuse — compound model: panel → judge → synthesize

Section titled “Q.fuse — compound model: panel → judge → synthesize”

Runs N specialist models in parallel on the same input, has a judge compare them, then a synthesizer writes one new answer none of the panel produced. The lift is the synthesis step, so small on-device models, judged and synthesized, can beat any one of them. Seals a holoq:FusionReceipt. Source: holo-q-fuse.js.

When to reach for it: you want the lift of diverse reasoning — panel members should span model families or personas.

What it refuses to fake

No panel configured → { ok: false, refused: true, reason: 'Q.fuse: no panel configured…' }. A panel of fewer than two members → refused (a panel needs ≥2 members (diversity is the point)). It never claims a synthesized answer it did not produce.

Q.recall — retrieval over your private corpus

Section titled “Q.recall — retrieval over your private corpus”

Semantic ⊕ lexical ⊕ graph-expanded search over your own indexed objects, with optional cited-answer synthesis. The retrieval core runs zero models and is deterministic; synthesize: true is the only step that calls one. Seals a holoq:RecallReceipt. Source: holo-q-recall.js.

When to reach for it: searching a user’s own notes, docs, or memories without touching the open web.

What it refuses to fake

Empty corpus → { ok: false, refused: true, reason: 'Q.recall: the corpus is empty — index pages first' }. synthesize: true with no synthesizer bound → answer: null (an honest null, never a fabricated answer).

Q.factory — signal → change → verify → seal → learn

Section titled “Q.factory — signal → change → verify → seal → learn”

The software factory as one verb: a red signal (a failing check, a bug, a request) becomes a located target, a proposed change, a verification, a sealed receipt, and a learned skill. Source: holo-mind-ui.js.

When to reach for it: Q.factory(signal) for a manual fix; Q.factory.watch(intent) for a hands-off loop a user has explicitly armed.

What it refuses to fake

With no verifier bound it returns outcome: 'unverified' — it never claims a fix it did not prove. If the conscience blocks, no attempt seals and no effect mints.

Q.hermes — native in-tab agent

Section titled “Q.hermes — native in-tab agent”

A native, serverless Hermes-compatible agent loop on the Q door (ADR-0100).

Status: design only

Q.hermes is specified but not yet implemented. The design declares the six terminal backends (Docker, SSH, Modal, Daytona, Singularity) incompatible in-tab and promises never to fake them — listed here so its absence is documented, not implied.
