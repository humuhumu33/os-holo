Hologram OS is a sovereign, serverless internet computer that boots entirely in
your browser. There is nothing to install, no account, and no server in the loop —
every object you touch is self-verifying, named by the hash of its content (Law L5).

This page walks you through your first session along whichever door fits you. If
you only want the mental model first, start with [What is Hologram OS?](/docs).

<Aside type="note" title="What you need">
  A modern browser is the whole requirement. No Node.js, no install, no account
  for booting. Building the OS or running a witness locally needs Node 20+ and Git.
</Aside>

There are three doors into the same substrate. Each ends at a command you can run
right now.

* **Build** — clone the repo, run the gate, and witness a component against an external authority.
* **Connect** — start the MCP door and discover its live, self-describing tools.
* **Boot** — open the gateway and use the desktop. No tools required.

## Build against it

You want a substrate where *done* is provable, not asserted. A component is
complete only when its conformance row is witnessed against an external authority
and the release gate stays green.

<Steps>

1. **Clone and enter the system root.** All work runs from `system/`.

   ```sh
   git clone <repo> && cd hologram-os/system
   npm install
   ```

2. **Run the release gate.** It re-runs each pure-Node witness live, joins it to
   the conformance catalog, fails closed on any unwitnessed required row, and emits
   a W3C EARL report. It must stay green.

   ```sh
   npm run gate
   ```

3. **Witness a single component** while you iterate. Each witness writes its own
   result object you can inspect.

   ```sh
   node tools/<name>-witness.mjs   # writes tools/<name>.result.json
   ```

</Steps>

<Aside type="caution" title="Mint nothing">
  Use W3C, schema.org, Dublin Core, PROV-O, and EARL terms. Never invent a private
  vocabulary where a standard term already exists.
</Aside>

## Connect an agent

You are an agent that must act without trusting a server. Every door hands back a
self-verifying object you re-derive before you trust it.

<Steps>

1. **Start the MCP door.**

   ```sh
   npx hologram-mcp
   ```

2. **List the live tools.** The door is self-describing — read the tool list rather
   than hard-coding this page.

   ```jsonc
   // list the live, self-describing tools
   { "method": "tools/list" }
   // e.g. verify_object · resolve_object · own_verify · own_settle · own_passport
   ```

3. **Re-derive every result.** Each object's identity is its hash; re-derive and
   compare (Law L5) before you act on it.

</Steps>

<Aside type="tip" title="The authoritative entry point">
  Start at the self-verifying discovery document
  [`/.well-known/agents.json`](/.well-known/agents.json). It lists every agent door
  — MCP, NANDA, A2A, Agent Skills — and is authoritative, not this page.
</Aside>

## Boot the OS

No tools required. Opening the gateway starts the κ Service Worker and walks the
boot chain on your own hardware — useful to see the substrate you build against.

<Steps>

1. **Open `index.html`** — the single gateway. It starts the κ Service Worker and
   enters the boot chain: rEFInd → Plymouth → SDDM → PrimeOS.

2. **Let it boot to the greeter.** The OS is content-addressed, so the bytes are
   verified on the way in — a single altered byte yields a different hash and is refused.

3. **Sign in to bind a device-local key,** then use the desktop. The key is bound
   to *this* device; there is never a server account to create.

</Steps>

<Aside type="tip" title="Deny the origin">
  The gateway is one CDN among peers, not an authority. Deny it and the
  byte-identical, verified OS still boots from cache or a neighbouring peer.
</Aside>

## Now what?

You're in. Here are the moves that pay off next.

**Verify, don't trust.** Every object names itself by `did:holo:sha256:H(canonical-form)`.
Re-derive the hash and compare before you rely on anything — the [κ-route](/docs/reference/kappa-route)
does this on every receipt.

**Read the Constitution.** Every action is checked against a fail-closed
[Constitution](/docs/concepts/conformance) that is itself an object you re-derive,
so a single altered byte in *it* is refused too.

**Learn the five Laws.** The whole system follows from [five Laws](/docs/concepts/the-five-laws);
L5 — verify by re-derivation — is the one you'll lean on most.

**Witness your work.** A component ships only when its [conformance row](/docs/reference/conformance)
is witnessed and green. Treat the gate as the definition of done, not a formality.

**Map the doors.** The [agent doors](/docs/reference/agent-doors) reference enumerates
every protocol surface and the verbs each one exposes.
