# ADR-0091: Q — the one unified surface for hologram-native intelligence: a single door every human, app, and external agent touches, composing the whole Holo Q spine behind one verb

**Status:** Accepted (the unified surface landed + Node-witnessed 17/17 + verified live in the booted shell). Ships at `system/os/usr/lib/holo/q/holo-q.js` (`createQ` / `installQ`), installed as `window.Q` by `holo-q-trinity-ui.js`, and the Create studio now builds through it.

**Context.** The reflection on the whole Holo Q stack (the move in [0087](0087-holo-trinity.md)/the spine work) left a real residue: even after the Create build was routed onto the trinity spine, *intelligence on the substrate was still a scatter of globals* — `HoloTrinity` (create·exist·perceive), `HoloMind` (the orchestrator), `HoloBoost` (the frontier tier), `HoloQOrchestrator`, `HoloConscience`. A human surface, an app, and an external agent each had to know which global to reach and in what order. That is the opposite of "abstract complexity, deliver simplicity," and it has a second, sharper problem: **there was no single, governed door for non-human callers** — an external agent would have to be handed interior objects (the trinity, the fabric), which is both confusing and a security hole. The objective: ONE surface, **Q**, that is the single point of access for hologram-native intelligence — for the user, for every native app, and for all external AI agents and other non-human entities — self-healing, ubiquitous, and honest about what it is.

**Decision.** Introduce **Q**: a thin, unified facade that *composes — never replaces —* the existing stack and exposes it as one calm object.

1. **One verb for humans, apps, and internal callers — `Q.create(intent)` / `Q.ask(intent)`.** It rides the spine: `Q.create` delegates to `trinity.create` over the Mixture-of-Specialists fabric ([0084](0084-holo-q-mux.md)/[0085](0085-holo-q-fabric.md)), so a build is content-addressed and a repeat ask is O(1) (the κ-memo), auto-perceived ([0086](0086-holo-perception.md)) with a real rebuild thunk (healable). `Q.ask` is the await form (the sealed value). The Create studio's `holoBuild` now calls `window.Q.create` — the studio and an agent take the *same* path.
2. **One GOVERNED door for external / non-human entities — `Q.agent(request, {caller})`.** This is the single point of contact for outside callers, and it is **fail-closed**: the conscience ([0033](0033-holo-constitution.md)) must accept, every accepted act seals a receipt (who · verb · the content-addressed effect κ), and **the interior is never handed out** — an external entity gets exactly this function, not the trinity/fabric/scene. With *no* conscience present an unknown external caller is **denied by default**; only an explicit host opt-in (`allowUngoverned`) lets an internal trusted caller through. The witness proves: no-gate → refused, conscience-blocks → refused, conscience-accepts → built through the same spine with a re-derivable receipt whose `effectKappa` *equals* the human-verb κ (one spine, one result).
3. **Ubiquitous, by one singleton — `installQ() → window.Q`.** Every native app inherits the *one* Q (the install pattern of `holo-mind-ui`/`holo-prov-ui`). Q holds the trinity/mux references internally; apps reach intelligence through `window.Q`, nothing else. **Ambient form (landed):** Q lives as an omnipresent **orb** (bottom-left) that expands into a light, non-blocking copilot panel — never a screen takeover; summonable four ways (orb click · `⌘I` · omnibar `⌘Enter` · an unhandled voice turn). It is **OS-aware** (`Q.scope(world, focusedId)` summarizes the whole desktop into every answer's context) and **context-aware** (`Q.create/ask({context})` resolves "this" against the focused holospace). It is also **omnipotent** — a pure router `Q.intent(text)` sorts a line into **act** (open · close · minimize · maximize), **ask**, or **build**, and the orb executes the action verbs against the OS's *existing* open/window primitives (mount an app, close/minimize/maximize the focused holospace) — so "open files", "close this", "go to github.com" just work, no new machinery. **Cross-frame + universal rollout (landed):** because a sandboxed app is a separate window that does NOT inherit `window.Q`, `holo-q-app.js` bridges it over the *same governed* `holo-privacy:rpc` channel `holo-gov.js` already brokers, and the **compositor auto-injects it into every same-origin app on mount** (one point — the iframe-load handler that already registers the frame — so all apps become Q-conformant with *no per-app edit*; cross-origin/web frames are excluded by design). The app gets `window.Q` + an in-app `⌘I` and reaches the shell's Q with the host un-forgeably asserting the frame's identity (`byWin`); `q.create` from an app rides Q's **governed door** (fail-closed + receipted), and the app frame holds nothing. Witnessed `holo-q-bridge-witness.mjs` 9/9 (mock bus: summon/ask/governed-create/refuse-on-block/host-asserted-identity/app-holds-nothing) + live (a real non-kernel app — Files — got `window.Q` injected and summoned the shell orb).
4. **Self-healing + bounded adaptation, labelled honestly.** `Q.perceive()` exposes the live scene (coherent/drift); `Q.improve()` / `startEvolving()` run the bounded, governed self-improvement loop ([0086](0086-holo-perception.md)). `Q.remember(signal)` / `Q.context` is a *small* per-user preference memory — recent intents (continuity, deduped) + 👍/👎 — used to bias routing/edit-context. The contract (`Q.describe()`) states plainly that "self-evolving" is the bounded heal/reevaluate loop, **not** open-ended learning, and "adapts to the user" is this preference signal — no theater.

**Honest costs / non-goals.** Q is a *facade*, not new intelligence — its power is exactly the spine's; the win is the single, coherent, governed door (the simplification). The external door (`Q.agent`) is a governed **function**; wiring it to a real network transport (MCP / `postMessage` / a relay) is a deliberate, separate, opt-in step — that is the actual attack surface and is **not** opened here. Adaptation is a preference bias, not a learner. A build is on-device/template unless a boost model is installed, in which case it is a governed network call ([0090](0090-holo-q-remote.md)) — Q inherits that exact trade, it does not erase it.

**On the "two κs" (a correction, after inspection).** An earlier note flagged the studio's *display* κ vs the fabric *compute* κ as a defect to unify. Inspection shows they are **two layers by design, not a bug**: the display κ (`repo.publishSource`) addresses the *published holospace document* — a JSON-LD `schema:WebApplication` over `{name, text:source}` — which is what the user shares and re-derives; the fabric κ addresses the *raw compute output value* (an internal memo + provenance key). The user already sees exactly **one** κ (the published one); the fabric κ never surfaces. Forcibly collapsing them is *wrong*: folding the compute κ into the published address (e.g. via `prov:wasDerivedFrom`) would make two identical holospaces built from different prompts get **different** addresses — a violation of Law L5 content-convergence (lineage is metadata; it must not change the content address, per [0088](0088-holo-home.md)). So Q keeps the two as distinct, correctly-layered addresses; an explicit compute→document provenance edge, if ever wanted, belongs in Holo Prov ([0082](0082-holo-prov.md)) as *separate* metadata, never in the content κ.

**Composition.** `holo-q.js` composes [0087](0087-holo-trinity.md) (the trinity it delegates to), [0084](0084-holo-q-mux.md)/[0085](0085-holo-q-fabric.md) (routing + κ-memo), [0086](0086-holo-perception.md) (perceive/heal), [0033](0033-holo-constitution.md) (the conscience gate on the external door), and inherits [0090](0090-holo-q-remote.md) for the governed boost tier. Grounded in Laws L1/L3/L5 (content address · O(1) replay · re-derivation). Mints nothing of its own (it returns the trinity's sealed κs + a lightweight provenance receipt on the agent door). Witnessed `system/tools/holo-q-unified-witness.mjs` 17/17 (emits `holo-q-unified-witness.result.json`, EARL `earl:TestResult`); the spine witness `holo-q-spine-witness.mjs` 12/12 (emits its result.json) covers the underlying path; live in the booted shell (`holo-os`/8300): `window.Q` installs and composes the trinity, `Q.create` cold→hot (repeat 1 ms), `Q.agent` routes through the real conscience + receipts, the studio build flows through `window.Q`. Pending: the external-transport binding (opt-in, the deliberate attack-surface step); the formal `#holo-q` gate conformance row and the os-closure re-lock (`holo-q.js`/`holo-q-trinity-ui.js`/`shell.html` κs moved) — both part of the deferred **branch-wide** seal batch (the gate is non-idempotent and the closure re-lock runs over the cross-repo `hologram-os` image, so neither is half-run per-change). The "unify the two κs" item is **withdrawn** — see the correction above (they are correct, distinct layers).
