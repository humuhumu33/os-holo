# ADR-0090: Holo Q Remote Model ‚Äî a non-local LLM is reached through ONE governed, witnessed substrate operation, not app-level fetch: the engine is a content address (never a URL), every answer mints a re-derivable receipt, and the conscience gates the egress

**Status:** Proposed (the egress seam, the boost subsumption, the cross-frame postMessage protocol, AND the live `shell.html` host-install all landed + Node-witnessed 31/31 at `system/tools/holo-q-remote-witness.mjs` + verified LIVE in the booted shell; a polished key-custody/consent UI, a `#holo-q-remote` conformance row, and the os-closure re-lock remain). The cross-frame layer: `holo-q-remote-client.mjs` (the app-side client ‚Äî a sandboxed holospace reaches the host broker over the SAME `holo-privacy:rpc` channel holding no key/URL), `serveRpc(broker,{app,method,args,onDelta})` + `listProviders()` in `holo-q-remote.mjs` (the host dispatch, where `app` is the host-asserted identity), and an additive delegation branch in `holo-gov.js` (`q.remote.*` ‚Üí `window.HoloQRemoteServe`, streaming `holo-privacy:delta`, fail-closed). The shell installs ONE host broker (`window.HoloQRemote` + `window.HoloQRemoteServe`) with the conscience resolved lazily; boost registers into that SAME broker (`createBoostSampler({broker})`), so the shell frame and every iframe app share a single governed vault. VERIFIED LIVE in the booted shell (`holo-os`/8300): the globals install with no console error, the conscience is sealed, and a full in-page round-trip with the real gate ran ‚Äî no-grant ‚Üí P4 block, granted ‚Üí streams "Paris." with a receipt that re-derives (tier "remote"), PII ‚Üí P5 block, and the vaulted key never appears in the result; `window.HoloQRemoteServe` denies an ungoverned frame and serves no-secret discovery for a host-asserted app. Node-witnessed too: provider discovery with no secret, a grant stamped to the host-asserted id, streamed deltas + a verifiable receipt, PII hard-blocked at the host, the key NEVER crossing the bus, an ungoverned frame denied, and boost driving the unchanged codegen seam through the shared broker. The existing `holo-app-governed-witness` still passes (the privacy broker is intact). The provable core ships at `system/os/usr/lib/holo/q/holo-q-receipt.mjs` (the InferenceReceipt minter ‚Äî RFC 8785 JCS + SHA-256, byte-identical to `holo-conscience.js`), `holo-q-adapter.mjs` (pure provider shaping for Anthropic ¬∑ OpenAI ¬∑ Ollama ‚Äî the any-llm value in three functions, zero dependencies), and `holo-q-remote.mjs` (the host broker: vault ¬∑ grants ¬∑ egress gate ¬∑ streamer ¬∑ receipt). **Boost subsumed** ‚Äî `holo-q-boost.js` is now the Anthropic specialization of the broker: its raw `fetch`/SSE moved to `adapters.anthropic`, the key moved off `window`/the sampler into the vault, and every generation passes the egress gate + mints a receipt (`.lastReceipt`); `window.HoloBoost` stays the same `(messages,opts)‚Üíasync-iterable` sampler, so `shell.html`/`holo-q-codegen.js` need NO edit. Witnessed in Node with a MOCK fetch and the REAL conscience: provider-agnostic SSE shaping + reconstruction for three wire formats; the engine identity is a content address `{wireFormat,modelId}` with no URL in it (Law L1); the response is content-addressed (identical bytes ‚Üí identical Œ∫, Law L3); every admitted answer is a re-derivable PROV-O `holoq:InferenceReceipt` and tampering any byte (even the tier label) is refused (Law L5); the conscience gates EGRESS ‚Äî PII in the outgoing prompt trips P5 (red line) ‚Üí HARD BLOCK, no grant ‚áí P4 block (ADR-0033); capabilities are default-deny, host-asserted, scoped, revocable; the API key NEVER appears in a grant, a receipt, or the app-facing result; and the subsumed boost drives the UNCHANGED codegen seam as mode "boost", mints a verifiable receipt, blocks PII at egress, and never exposes the vaulted key.

**Context.** The question that opened this: is it worth vendoring a unified-LLM gateway (e.g. mozilla-ai/any-llm) to give Holo Q "seamless access to non-local LLMs"? No ‚Äî it is Python, it routes to remote servers, and "seamless/frictionless" is exactly the wrong default for a substrate whose value is that it does not phone home. But the real finding is sharper: **boost as it ships today (`holo-q-boost.js`, ADR-0087) is already the Law violation the gateway would formalize.** It is a bare `window.HoloBoost` doing raw `fetch` to `api.anthropic.com` from app/page scope ‚Äî a parallel network (breaks L4 "no parallel network"), an engine named by URL (breaks L1 "nothing is identified by host, path, or URL"), with the key visible to any script on the page and no audit trail. The apparent objection to a remote model is L5 (non-reproducible bytes can't be re-derived) ‚Äî but that objection is false by inspection: the DEVICE tier (Qwen2.5 with any sampling) is ALSO non-reproducible, yet nobody claims it breaks L5, because L5 is *"re-derive every received byte against its Œ∫ before accepting it"* ‚Äî byte‚ÜíŒ∫ verification, NOT function reproducibility. A remote model is identical on that axis. **The only real delta a remote call adds is L1/L4 egress** ‚Äî and that is a governance problem, not a determinism one. So this ADR does not add a remote capability beside the Laws; it brings the one egress the OS already performs UNDER them.

**Decision.** Make a non-local LLM reachable only as a single, governed, witnessed substrate operation, with the boundary object ‚Äî the receipt ‚Äî as the thing that crosses into the store:

1. **The engine is a content address, never a URL (L1).** A provider is a descriptor `{wireFormat, modelId}` whose Œ∫ (`providerKappa`, JCS+SHA-256) is its identity ‚Äî shareable, re-derivable, URL-free. The endpoint URL + API key live ONLY in the host **vault** (`holo-q-remote.mjs`), bound to that Œ∫ out-of-band, never content-addressed, never witnessed, never posted to an app frame. This is how a Œ∫-disk binds to GPU memory: the substrate holds the Œ∫; the driver resolves it. The witness proves no URL leaks into the engine identity and the key never leaves the vault.
2. **Egress is ONE brokered operation, not app fetch (L4).** `infer({app, providerK, messages})` is the single seam; the network is ingest-boundary I/O (like a camera), not a parallel runtime ‚Äî the remote weights/runtime never join the address space, only the receipt Œ∫ does. This is the precise repair of boost's L4 hole. The pure `runSSE` reconstruction logic the witness exercises is the SAME logic the live `streamRemote` reuses over the fetch reader.
3. **Every answer mints a MANDATORY re-derivable receipt (L5 + P2).** The response bytes are content-addressed (`responseKappa` ‚Äî identical generations converge to one Œ∫, L3), and a `holoq:InferenceReceipt` (PROV-O Activity) seals `{request Œ∫, response Œ∫, engine Œ∫, usage, cost, timing, conscience verdict, host-asserted app}` to a `did:holo`. `verifyReceipt` re-derives it in-tab; a single altered byte is refused. It is not optional: P2 (Provenance duty, ADR-0033) makes "no receipt" a `leavesNoAuditTrace` violation, so a remote answer with no receipt is never admitted. The receipt slots into Holo Prov (`walkLineage`/`reDeriveClosure`, ADR-0082) ‚Äî a remote-assisted artifact commits the receipt into its closure and carries honest "remote-assisted" lineage forever.
4. **The conscience gates the EGRESS (ADR-0033).** Before any send, the OUTGOING prompt is mapped to the world-vars and run through the EXISTING fail-closed gate: **P5 Data minimisation (RED LINE)** ‚Äî PII in the prompt without lawful disclosure ‚Üí HARD BLOCK (private data does not leave the device, no caveat); **P6 Classification handling (RED LINE)** ‚Äî classified context ‚Üí block; **P4 Consent** ‚Äî no grant ‚áí `breachesConsent` ‚áí block (the grant IS the consent envelope). A non-red-line caveat is allowed-but-watermarked: the artifact's manifest carries the receipt Œ∫ and the prov badge shows the remote tier, so it never lies about which engine ran (the codegen honesty contract, ADR-0087).
5. **Capabilities are default-deny, host-asserted, revocable.** `requestCapability` mints a grant only for a registered provider and stamps the recipient from the HOST-asserted identity (the `holo-gov.js` `byWin` pattern ‚Äî un-forgeable, an app cannot request as someone else), scoped (token budget ¬∑ rate ¬∑ expiry) and revocable. An app that never requests the capability keeps the "0 network calls in the core loop" guarantee (`studio-demo.html`'s fetch-interceptor) ‚Äî the broker simply never grants, and nothing in the Create loop calls it.

**Subsuming boost (landed).** `holo-q-boost.js`'s `streamClaude` raw-fetch/SSE is gone ‚Äî that logic now lives in `adapters.anthropic`. Boost is rewritten as `createBoostSampler(...)`, which builds a broker, registers the Anthropic engine in the vault (the key enters the vault here and never leaves it), mints a grant, and returns a sampler that bridges `broker.infer`'s `onDelta` + final receipt into the codegen contract `(messages, opts) ‚Üí async-iterable of text deltas`. `window.HoloBoost` is KEPT ‚Äî `shell.html` and `holo-q-codegen.js` read it unchanged and a boost build is still labeled mode "boost" ‚Äî but its NATURE changed: the key moved off the global into the broker's closure vault (the "key visible to any script" hole closed in-frame), every generation passes the conscience egress gate and mints a re-derivable receipt (exposed as `.lastReceipt`), and a PII prompt is hard-blocked at P5 before a byte leaves. `installBoost` is now fail-closed: it refuses to enable boost unless `window.HoloConscience` is present and sealed. Boost the bare network call became boost the governed capability, with zero edits to its consumers. (The full cross-frame "app frame holds nothing" guarantee ‚Äî the key in a separate host origin, unreachable even in principle ‚Äî lands with the postMessage broker; in the single-frame shell the closure vault is the honest containment.)

**Honest costs.** The receipt re-derives the bytes you received and proves a gated egress happened with exactly these request/response bytes under a sealed conscience ‚Äî it does NOT prove the bytes are genuinely from the named model (that rests on TLS + host integrity, both outside the substrate, exactly as the DEVICE tier trusts the Œ∫-disk it loaded); where a provider returns a signed/attested response it can be pinned into the receipt and verified (a strengthening, not a precondition). This is strictly MORE honest than boost today, which asserts the same engine with no receipt at all. Streaming over postMessage adds marginal latency (mitigated by a transferred MessagePort; network dominates first-token). The key still lives in the browser ‚Äî this design CONTAINS it (host-only, app holds nothing) rather than eliminating it; a build through the remote tier is not serverless, the same deliberate trade boost's header already names. And the postMessage broker wiring, the Create-mode subsumption, the consent/key UI, and the os-closure re-lock are the staged remaining work.

**Composition.** Built on the Holo Constitution (ADR-0033, the egress gate + P2/P4/P5/P6), Holo Prov (ADR-0082, the receipt's lineage), Host Governance (`holo-gov.js`, the un-forgeable host-asserted recipient), the Holo Q codegen seam (ADR-0087, the unchanged sampler contract), and the Œ∫-substrate (0052/0024/0025); grounded in Laws L1/L2/L3/L4/L5. Generalizes ADR-0087's boost from a bare network call into the governed, witnessed, provider-agnostic remote tier.
