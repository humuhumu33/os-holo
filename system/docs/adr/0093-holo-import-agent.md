# ADR-0093: Holo Import → agent-native — the moment a GitHub repo imports (ADR-0092) it is auto-WIRED into the MCP + /~<app>/api REST seams every holospace ALREADY exposes, so any AI agent or holo app can fetch · re-derive (L5) · run · share · discover it with zero author effort; Railway's "connect a repo → it runs" feel, zero servers, agent-native by default

**Status:** Proposed (agent-surface derivation + baseline loop + LIVE in-page auto-registration + DECLARED-surface tier + SERVERLESS TRANSPORT BRIDGE (external HTTP agent reaches an import via the SW) + FORGE BUNDLER CLASS (client-bundle repo → one self-contained app via a real esbuild κ-transform) + VENDORED κ-pinned esbuild toolchain (local-first, no CDN) + GOVERNED INGEST (conscience-gated, Law L4) all landed + Node-witnessed 45/45 agent · 38/38 import · 12/12 forge · 6/6 vendor against the REAL cores/esbuild/conscience, AND verified live in the booted shell with real repos, real esbuild-wasm (vendored + CDN), and the PII gate; OpenAPI-YAML + postMessage + governed-egress execution + non-esbuild build-plugin repos staged). Ships at `system/os/usr/lib/holo/holo-import-agent.mjs` (pure, isomorphic, deps injected); composes — never rebuilds — `holo-mcp.mjs`/`holo-mcp-core.mjs` (`buildAppRegistry`/`makeServer`/`handle`/`capabilityCard`) and `holo-api-core.mjs` (`makeApiServer`). The shell (`shell.html`) wires it: on a successful import, `registerLiveAgent(li)` registers the app into `window.HoloImports` (a `createLiveRegistry` over the node-free engines) AND `publishImportToSW(surf)` writes its manifest + self-verifying κ-objects into the `holo-imports-v1` cache the Service Worker reads — so the **already-existing** SW MCP/REST endpoints (`holo-fhs-sw.js`) answer `/~<id>/mcp` + `/~<id>/api` for the in-memory import to **any external HTTP agent**, with **no reload, no origin server**. A prerequisite Buffer-free `holo-object.address()` (byte-identical: `sha256hex(jcs(content))`) makes verify/L5 run in the browser + Service Worker.

**Context.** ADR-0092 made "paste a GitHub URL → a running, content-addressed Holo app" real for the static/SPA class. The next demand: that imported app must be **instantly drivable by AI agents and other holo apps** — Railway's promise (connect a repo and it's *live, reachable, networked*) but 100% serverless and agent-native. The finding that makes this small rather than a platform: **the endpoints already exist.** Every holospace already exposes (a) an MCP server generated from an `apps/<id>/holospace.json` manifest (`holo-mcp.mjs`), where every resource is a self-verifying UOR object and a standardized capability card (`holo://capabilities`) describes the surface; and (b) a unified REST surface at `/~<app>/api` (`holo-api-core.mjs`) — content-addressed κ-object ingress/egress, HTTP-402-payable, **mounted on every tier including the Service Worker** (serverless by construction). Both take the *same* `resolve(uri) → self-verifying object` over one κ-store. And an imported app's source IS already such an object — the `publishSource` envelope verifies under `holo-object.verify` (proven). So "agent-native" is not new infrastructure; it is **deriving the one manifest + the one resolver** that mount an imported app into seams that already work.

**Decision.** Auto-derive the agent surface on import; expose, never auto-authorize.

1. **Baseline (every imported app, free).** `deriveManifest(imp)` emits a `holospace.json` declaring the app's source as a self-verifying **resource** (uri = the app κ); `deriveAgentSurface(imp)` returns that manifest + a `resolve(uri)→object` + a NANDA AgentFacts record. Fed to `buildAppRegistry` + `makeApiServer`, the instant the app imports an agent can: **list** it (`tools/list`, `resources/list`, app-scoped server name), **fetch + re-derive** it (`resources/read`, `GET /~<id>/api/o/<κ>` with `x-holo-verify`, Law L5), **run/share/inspect** it (the universal built-ins `resolve_object`·`verify_object`·`holo_describe`·`holo_run`·`holo_share`), and be **discovered** (the AgentFacts record). No author writes anything.
2. **One resolver, two surfaces, no parallel infra.** The exact same `resolve` is fed to the MCP `handle` ctx and to `makeApiServer` — one κ-store, MCP and REST are two projections of it (witnessed: `ctx.resolve === api.opts.resolve`). The whole surface is isomorphic, so it runs in the Service Worker — no origin server in the path (the L4/serverless guarantee carries over from `holo-api`).
3. **EXPOSURE ≠ AUTHORIZATION (the security line).** The manifest *exposes* a surface; it grants **no capability**. Baseline `tools` is empty — the agent verbs are the universal, read-only/forge built-ins over the app κ, not app privileges; any effectful capability stays **default-deny behind the conscience** (ADR-0033). Auto-mounting an untrusted repo therefore hands out nothing but the ability to *fetch and verify content that already self-verifies*. This is the one place a careless "auto-API" would be a hole; it is closed by construction.
4. **DECLARED surface (when the repo HAS one — LANDED for OpenAPI JSON + typed exports).** `detectDeclaredSurface(files)` reads the repo: an **OpenAPI/Swagger JSON** spec → one MCP tool per operation, schemas **faithful** (query/path params + `requestBody` → a correct JSON-Schema `inputSchema`, names/descriptions preserved); **typed exports** (a `types`/`.d.ts` + the entry's named exports) → descriptive tools. The tools land in the manifest's `tools[]`, so `tools/list` + the capability card advertise them and an agent **discovers the app's real API**. Crucially each tool carries an honest **`x-holo-exec`**: a backend op is `egress` (needs a governed `holo-gov` grant, ADR-0090 — *not* serverless), an arbitrary export is `sandbox` (the conscience-gated sandbox, staged). `execDeclaredHandler` does **not** run either, so calling a projected tool without a wired/granted handler returns an honest error — **discovery, never authorization**. OpenAPI-YAML (zero-dep, no parser), `postMessage`, and the actual governed-egress/sandbox execution are the staged sub-steps.

**Railway → Hologram architecture map** (each Railway pillar → its 100%-serverless, content-addressed equivalent, with the honest gap):

|---|---|---|---|

| "It just works" + usage pricing | bills compute + egress + storage | same promise minus servers; **the 0-network core loop is free**, only egress / paid κ-streams 402-metered (`holo-api` already does HTTP 402) | honest: a *remote* model or a backend that needs egress is not free and not serverless (0090's named trade) |

**Honest costs / non-goals.** **"Fully accessible by agents" is true for the baseline κ-verbs and for repos with a DECLARED interface — not for an arbitrary app's DOM/internal functions.** Those do not become a clean tool API for free; the bounded fallback (a sandboxed, conscience-gated invoke) and what wasn't auto-exposed must be **surfaced, never claimed**. The managed-DB analogy is where "serverless" genuinely ends (see the table). This ADR lands the *derivation* + the proven loop; **live auto-registration** — emitting the manifest into the running shell's MCP server and the SW's `/~<app>/api` mount on import — is the staged next step (the Node witness mounts the real cores; the live SW wiring is the deliberate, separate step, twin of how 0090/0091 staged their live host-install). AgentFacts is a discovery record; a live NANDA registry publish is staged.
