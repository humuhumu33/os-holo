# ADR-0089: Holo Scene — the general composed-holospace manifest: any κ-app or experience that arranges κ-objects is ONE κ with render · link · remix · share · provenance over its LIVE STATE, not just its source; the desktop (ADR-0088) is its first specialization

**Status:** Accepted (the general engine landed + Node-witnessed 10/10; `holo-home-scene` refactored onto it). Ships at `system/os/usr/lib/holo/holo-scene.mjs` (pure, dependency-free). It generalizes the desktop scene-manifest (ADR-0088) so ANY composed holospace — an app that arranges several κ-objects, a multi-pane experience, a dashboard — declares its own scene the same way and inherits the five verbs over its **live state**: a manifest of `@type` (the scene kind) ⊕ `background` (an opaque κ-ref/seed) ⊕ `objects[]` (each `{k,x,y,w,h,z?,state?}` — ref + layout + a defined minimal round-trip state) ⊕ `props` (config κs), sealed to the scene κ. `render(manifest) ⇄ serializeWorld(live)` are **inverse** (the Atlas isomorphism), and `defineScene({type})` lets an app declare its kind once. `holo-home-scene.mjs` is now a thin specialization (the home is a `holo:HomeScene`: wallpaper-seed background + desktop objects + dock/theme props), with no behavior lost. The first per-object **state schema** ships at `holo-scene-state.mjs` — `holo:widget`, grounded in the Holo Widgets runtime (whose `serialize()` already persists `{type, config, hidden}` and keeps out the transient `_state`): a widget board round-trips its live state end-to-end (Node-witnessed 8/8 — board → scene → render → serialize → the same κ; a transient countdown tick changes nothing; a config edit → a new κ; remix → a child κ with provenance). Witnessed in Node 10/10: an app declares its own kind and seals to a κ; re-derivable order-independent; **`render → serialize` round-trips a composed app — including its live object state — to the same κ**; remix → a child κ linked `prov:wasDerivedFrom` (Holo Prov, ADR-0082); `walkLineage` to genesis; the home still seals and round-trips through the shared engine.

**Context.** The question this answers: does the desktop-as-holospace work (ADR-0088) apply to every κ-app that builds, runs, and is shared in an isolated holospace? Two facts had to be separated. First, the **five verbs are already universal** — they follow from content-addressing, so every κ-app already renders, links, remixes (of its source), shares, and tracks lineage; ADR-0088 only closed the one holospace (the desktop) that wasn't a document. Second, there is a real line between an app's **source** and its **running state**: `source κ → screen` is always deterministic and re-derivable, but a *running* app's live DOM ≠ its source — runtime state diverges, so you can remix its source exactly but cannot snapshot its live state into a κ *unless the app models that state as content*. The desktop's scene manifest was exactly that discipline. Generalizing it gives every composed holospace the same tool: a way to pull its **runtime across the re-derivable line**, so remix/share/provenance apply to the live thing, not just the file.

**Decision.** Make the scene manifest a first-class, reusable primitive any composed holospace declares:

1. **One engine, declared per kind.** `holo-scene.mjs` is the general model; `defineScene({ type, props? })` mints a factory so an app declares its kind once (`const AppScene = defineScene({type:"holo:AppScene"})`) and builds manifests ergonomically. The desktop becomes the first specialization (`holo:HomeScene`), not a special case — `holo-home-scene.mjs` re-expresses wallpaper→`background.seed` and dock/theme→`props` over the shared engine, losing nothing.
2. **Live state made re-derivable (the new capability).** `objects[]` carry an optional `state` — a *defined minimal* round-trip state per object kind. `render(manifest) ⇄ serializeWorld(live)` are inverse, witnessed by a composed app round-tripping (live object state included) to the identical κ. So an app that opts into a scene manifest can seal its **running state** to a κ: share the running app's exact state, remix from it, and walk its lineage — the runtime is now content. An app that doesn't opt in still has all five verbs over its **source** (the default, unchanged).
3. **Uniform identity + lineage.** `sceneKappa` addresses the content (kind ⊕ background ⊕ objects ⊕ props), so the same composed scene re-derives to the same κ everywhere (Law L5, order-independent); `remix(parent, edit)` derives a child carrying `prov:wasDerivedFrom`; `walkLineage` walks to genesis. The same lineage hypergraph holds apps, experiences, and the home.
4. **Isolation preserved.** A scene references κ-objects; it does not break their sandboxes. Rendering a scene instantiates each object (each can stay isolated); the manifest is the *composition*, not a hole in the boundary. So an app that builds/runs/shares in an isolated holospace stays isolated — the scene κ operates on references and layout, never on the running sandbox.

**Honest costs.** `background` is a **ref/seed, not pixels** — content-exact, not frame-exact. `objects[].state` is a **defined minimal set named per object kind** (what a chart persists vs. discards), not "whatever the live DOM holds" — naming that set is the design work each composed app owes; an undisciplined dump is not re-derivable. Opting in is the app's choice: a naive app remixes its *source*, a scene-declaring app remixes its *state* — both are first-class, but only the second makes runtime re-derivable, and only as faithfully as its declared state. And a scene that references a since-removed object κ re-derives to a missing tile — surfaced, not hidden.

**Composition.** Generalizes ADR-0088 (the desktop, now its first specialization); composes Holo Prov (ADR-0082, lineage), Holo Cosmos (ADR-0080, the home's seed background), the Atlas isomorphism (ADR-0043), and Law L1/L5. Mints only `holo:Scene`/scene-kind types over schema.org/PROV-O. Landed the first state schema (`holo:widget`); pending: state schemas for the remaining app/object kinds, a `#holo-scene` witness row, and the compositor/Create wiring (shared with ADR-0088's staged refactor — now the home and any composed app ride the same path).
