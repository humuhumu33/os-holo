# ADR-0095: Holo DevTools — the F12 developer environment, native to the κ-substrate and equally usable by humans, AI agents, and Holo Q: vendor the Chrome DevTools frontend UNMODIFIED, build the κ-CDP backend

**Status:** Proposed — **DESIGN ONLY** (this ADR + the CDP↔κ flow-map at [`../holo-devtools-cdp-flowmap.md`](../holo-devtools-cdp-flowmap.md); **no app code**, awaiting acceptance per the brief's deliverable 4). The seven-stage build plan, vendoring recipe, and per-stage witnesses are specified, not landed.

**Context.** Every other holospace pillar is built — Build·Run·Share ([0055](0055-holo-build.md)), language intelligence ([0056](0056-holo-lang.md)), the scene model ([0089](0089-holo-scene.md)), the unified intelligence door ([0091](0091-holo-q.md)) — but there is no way to *inspect* a running holospace the way a developer inspects a web page with F12. A κ-app renders a live scene from a manifest, streams κ-addressed objects, runs Q transforms, and exposes MCP/REST seams — and all of it is currently opaque at runtime. You can read the source ([0055](0055-holo-build.md)) and the lineage ([0082](0082-holo-prov.md)), but you cannot open a panel and see the live object tree, the resolved design tokens, the κ-stream timeline, or the heap of the object store. Worse, the same gap exists for the OTHER two audiences: an AI agent debugging an imported app ([0093](0093-holo-import-agent.md)) and Holo Q reasoning about why a widget won't render both lack a first-class inspection surface. The brief asks for the F12 experience, native to the substrate, equally usable by humans, external AI agents, and Holo Q, very low-latency, making the most of κ-addressed streamed+rendered objects.

Two off-the-shelf options were named: Mozilla's Firefox Developer Edition, and `ChromeDevTools/devtools-frontend`. They are not the same kind of thing. Firefox Developer Edition is a whole BROWSER with a built-in inspector that has no embeddable, hostable frontend — you cannot mount it inside a holospace or point it at a non-Firefox backend. `devtools-frontend` is the OPPOSITE: a standalone web app that is a pure client of the Chrome DevTools Protocol (CDP) over a swappable `Connection`, with no assumption that it is talking to Chrome. It assumes a CDP peer, not a browser. That is exactly the shape the substrate needs.

**Decision.** Adopt `ChromeDevTools/devtools-frontend` as the UI shell — vendored UNMODIFIED at a pinned commit — and build the load-bearing piece: a **κ-CDP backend** whose inspected objects ARE κ. Firefox Developer Edition is **rejected** (justification below). The UI is the cheap part; the backend that maps CDP ⇄ the κ substrate is the engineering, and it is built ONCE and reached through three doors.

1. **Reject Firefox, adopt the Chrome DevTools frontend — and own only the backend.** Firefox Developer Edition cannot be embedded as a holospace and cannot be pointed at the substrate; it is a browser, not an inspector you can host. `devtools-frontend` renders the familiar panels (Elements · Styles · Console · Sources · Network · Performance · Memory · Application) and speaks CDP over a `Connection` we control. We vendor it byte-for-byte (Law L5: a tampered DevTools asset is refused on read by `holo-fhs-sw.js`) and write ZERO UI. The only first-party code is a small adapter that boots the frontend and binds its transport, plus the κ-CDP backend.

2. **A κ-CDP backend — every inspected handle is an alias for a κ (NOT a port of Chrome).** We implement a FAITHFUL SUBSET of CDP domains where every `nodeId`/`objectId`/`scriptId`/`requestId` is a stable, content-derived alias for a κ. **DOM** ⇄ the scene manifest tree ([0089](0089-holo-scene.md)/[0088](0088-holo-home.md)), each node's `backendNodeId` ⇄ its κ, children = nested κ-objects (the Merkle-DAG). **CSS** ⇄ the resolved `--holo-*` design tokens ([0023](0023-holo-theme.md)/[0030](0030-holo-ui.md)) with the readability floor ([0057](0057-holo-ui-conformance.md)) shown as an un-overridable rule. **Sources** ⇄ the source κ (`publishSource` → `SoftwareSourceCode`, [0055](0055-holo-build.md)/[0056](0056-holo-lang.md)). **Network** is the DIFFERENTIATOR — not HTTP but the κ-addressed streamed-object fetch/render timeline: per "request" we show the κ, the axis (sha256/blake3), the **cache hit** (an O(1) `Map.get`, Law L3), the **L5 verify badge** (`verifyDeep` pass/fail — a tampered κ shows red), the render cost, and the **stream provenance** (`prov:wasDerivedFrom`). **Memory** ⇄ the κ store itself (heap node = κ-object, edges = `links[]`, dedup visible, [0060](0060-holo-link.md)). Reads are read-through-κ (`resolve` + `verifyDeep`); writes (editing a node) go through the `holo-blocks-repo`/scene write path (`handle.change` → `remix` child + `prov:wasDerivedFrom`), lineage preserved. The full per-domain table is the flow-map ([`../holo-devtools-cdp-flowmap.md`](../holo-devtools-cdp-flowmap.md) §2).

3. **THREE doors, ONE backend (mirror [0091](0091-holo-q.md)).** All three entries call the SAME `HoloDevToolsServe` dispatcher; the door changes only the transport and the governance posture, never the logic. **Humans** get the vendored frontend mounted as a right-dockable/full-screen holospace, speaking CDP over the cross-frame channel. **AI agents** get the SAME CDP methods projected as governed MCP tools (`devtools_inspect`·`devtools_query_dom`·`devtools_network`·`devtools_eval`) plus a CDP-over-REST seam at `/~holo-devtools/api` (the [0093](0093-holo-import-agent.md) wiring), each returning a self-verifying κ result (verify, don't trust — the [0049](0049-holo-agents.md) pattern), **fail-closed**. **Holo Q** gets a first-class verb so it can inspect and mutate the running scene through the same governed path, κ-memoized and deterministic. Because every handle is a κ (content-derived), the three doors share one backend with no forked logic.

4. **Transport is in-process / cross-frame — the low-latency choice.** CDP's default WebSocket round-trip is NOT used. The frontend's `Connection` binds to the EXISTING Q cross-frame channel (`holo-privacy:rpc`/`:res`/`:delta`, brokered by `holo-gov.js` `byWin`, the same bus [0091](0091-holo-q.md)/[0090](0090-holo-q-remote.md) run): one CDP message = one `rpc` with `method:"cdp"`; CDP events ride the existing `:delta` push; the caller's identity is host-asserted by `byWin` (un-forgeable). No socket, no localhost server, no JSON-over-TCP. A repeat inspection of an unchanged κ is an O(1) `Map.get` — **the κ-memo IS the DevTools cache** (Law L3) — and we stream deltas, never whole trees. `holo-gov.js` gains exactly one branch (`d.method === "cdp"` → `HoloDevToolsServe`), mirroring `HoloQServe`/`HoloQRemoteServe`.

5. **Governance — read your own scene sovereignly, gate everything else (Law L4).** Read-through-κ inspection of your OWN focused holospace is sovereign (re-derivation of content you hold, Law L5, like opening Holo Files). Beyond that, fail-closed through the conscience ([0033](0033-holo-constitution.md)): `Runtime.evaluate` and any mutation route through `Q.agent` (governed, receipted — never a raw eval, and the interior store is never handed out); cross-holospace inspection needs a default-deny grant; remote/off-device inspection rides the governed egress door ([0090](0090-holo-q-remote.md)) and opening a network transport for it is a deliberate, separate opt-in (the real attack surface — **NOT** lit here). Every admitted privileged op seals a re-derivable receipt (who · verb · effect κ · at). DevTools mints nothing: it returns the substrate's existing κs plus a lightweight inspection receipt on the governed doors.

**Strict Laws & W3C conformance — and the conformance centre of gravity.** The design conforms to the five engine Laws by construction: every handle is a content κ (**L1**), no new identifier scheme is minted — only `did:holo` ⊕ the BLAKE3 axis (**L2**), the κ-memo IS the cache so a repeat inspection is O(1) (**L3** — this is the "very fast in any browser" property, derived from the Law not bolted on), DevTools mints nothing and stands up no new authority (**L4**), and every read is `resolve`+`verifyDeep` with a tampered κ refused (**L5**). But "100% W3C compatible" must be claimed honestly, because two things genuinely strain it. **(T1) CDP is not a W3C standard and the vendored frontend bundle is not a JSON-LD object.** Resolution: **CDP is the HUMAN door's PRIVATE internal transport — agents and Q NEVER speak it.** The frontend is a byte-pinned vendored artifact (κ-addressed L1, SRI-verified L5), exactly like Medusa/JupyterLite/the voice model — not required to *be* semantic-web-native. The OPEN, interoperable surface — the one that must be 100% W3C for the open semantic web — is the **agent/Q doors**, and they are MCP ⊕ OpenAPI 3.1 (the REST seam) ⊕ NANDA/A2A discovery, returning self-verifying JSON-LD/UOR/PROV-O objects canonicalized by RFC 8785 JCS, addressed by DID+multihash, integrity-checked by SRI, witnessed in EARL 1.0 ([0022](0022-w3c-addressing.md)/[0024](0024-w3c-conformance.md)/[0025](0025-uor-object.md)/[0031](0031-holo-conform.md)). So the semantic web sees standards-native doors and self-verifying objects; CDP stays an implementation detail. The consequence for the build: **the agent door is a first-class W3C door with its own OpenAPI/MCP contract and its own witness, not a thin projection of CDP** — it shares the backend but its conformance is proven independently. **(T2) Accessibility / the readability floor ([0057](0057-holo-ui-conformance.md)) — i.e. W3C WCAG + WAI-ARIA** — an unmodified Chrome UI hardcodes sub-16px fonts and its own focus/contrast model. Resolution (to ratify): a holo-theme token shim at the mount boundary where the bundle's CSS permits, else the verbatim-vendor exemption the `#holo-ui-conformance` witness already grants SDDM/Plymouth — declared explicitly in the `#holo-devtools-vendor` row, never implicit. The flow-map ([`../holo-devtools-cdp-flowmap.md`](../holo-devtools-cdp-flowmap.md) §7) carries the full Law-by-Law and spec-by-spec table; a dedicated `#holo-devtools-conform` row asserts T1 mechanically (canonical, `verifyDeep`-able JSON-LD out the agent door, zero CDP leakage, OpenAPI+NANDA facts re-derive) — that row is what backs the 100%-W3C claim.

**Firefox vs Chrome — the decision named.** The brief pre-decided Chrome; inspection confirms it is the only viable option, for a structural reason, not a preference. Firefox Developer Edition ships an inspector welded into a full browser — there is no separable, hostable frontend artifact to mount in a holospace and no documented protocol seam to repoint at a foreign backend. `devtools-frontend` is a hostable web app whose entire contract is "be a CDP client over a `Connection`". The substrate already has a content-addressed object graph, a render⇄serialize isomorphism ([0089](0089-holo-scene.md)), a streamed-object model, and a cross-frame governed bus — i.e. it can BE a CDP peer with a thin backend. So the choice is forced: the option that separates UI from protocol wins, because the substrate's value is in the backend mapping, and only Chrome's frontend lets us own that backend while inheriting a familiar, complete UI for free.

**Honest costs / non-goals.** This is a CDP SUBSET, not the whole protocol — the mapped domains (flow-map §2) are faithful; unmapped methods return a clean "unsupported" and those panels stay inert (graceful degradation, not a lie). **No live V8 stepping in phase 1**: Sources maps to κ source OBJECTS, not a real `Debugger` over the page's running JS (breakpoints/stepping/scope/watch); Holo Build/Lang already own the editor, and true step-debugging of a holospace's JS is a later, separate stage. The Performance panel shows substrate timings (resolve/verify/render), not a V8 sampling profiler. `devtools-frontend` is a LARGE bundle — vendoring it unmodified accepts that weight. **Correction after vendoring (the honesty bar):** npm's `chrome-devtools-frontend` ships the TypeScript SOURCE tree, NOT a runnable build (no compiled `inspector.js`; browsers can't run `.ts`), and Google's old pre-built CDN is dead. The transport binding is proven against that source (the embedder seam is read from `Connections.ts`), but a BOOTABLE bundle must come from a local Chrome/Edge install (this box has Chrome 149, which serves the built devtools) or a from-source GN/Ninja build — see the flow-map §5. So "vendor the built bundle" required a corrected sourcing step; it did not invalidate the architecture (both Node witnesses are green), it relocated where the built bytes come from. Network/Memory/Performance are RE-INTERPRETED onto the substrate's reality (κ streams · the object store · re-derivation cost) — that is the whole point, but the numbers mean substrate things, not browser things, and the UI copy must say so. Remote cross-device inspection is gated and **unlit**. And like every recent ADR, the os-closure re-lock + branch-wide seal is a deferred maintainer batch (the gate is non-idempotent), not half-run per stage.
